Q1: FoundationDB seperates its architechture into a TMS and SS so that each part can focus on its different duties. TMS handles things like ordering transactions, assigning versions, and detecting conflicts, and SS handles storing the actual data and doing the read requests. The speration makes the system easier to scale and also improves performance because each component can be optimized by itself instead of doing everything at once.

Q2: During a write transaction in FDB, read write conflicts are addressed by the transaction management during the commit phase of the transaction. Right before a transaction commits the system checks whether any data it read before has been motified by another transaction. If there is a conflict, the transaction is aborted and it has to be retried. The makes sure that transactions do not happen on inconsistent data.

Q3: The purpose of the read version in FoundationDB is to give each transaction a consistent snapshot of the database at a certain point in time. When a transaction begins it is assigned a read version and all reads within that transaction see the database exactly as it existed in that certain moment. This makes sure that the transaction happens on a stable view of the data which is good for helping to prevent inconsistencies.

Q4: Snapshot reads differ from standard reads with respect to conflict detection because they do not take part in conflict detection. Standard reads might cause a transaction to retry if the data changes while snapshot reads ignore these conflicts and are more effecient. This means that snapshot reads are useful when performance matters more than correctness while standard reads are used when consistency is the most important.

Q5: FDBs recovery is cheap since it does not need to replay a full redo log. StorageServers hold most of the data when LogServers keep the most recent updates. After a failure the system recovers by combining these and avoides slow full log replay and allows fast recovery.
